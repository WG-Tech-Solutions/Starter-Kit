Hardware Requirements

The WGtech AI Accelerator Starter Pack comes as a complete unit. Additionally, you will need the following components:

ComponentNotes
WGtech AI Accelerator UnitRaspberry Pi 5 with pre-installed Axelera Metis M.2 PCIe card on PCIe HAT — supplied as one integrated unit

✅
Starter Pack Advantage The Raspberry Pi 5 and Axelera Metis card come pre-assembled and tested as a single unit. The Metis card is already properly seated in the M.2 slot with the PCIe HAT correctly stacked and secured — no physical assembly required.
🔧 Debug: Physical Installation Notes

If the card is not being detected or you need to reseat it:

The Axelera Metis card must be seated firmly in the M.2 (M-key) slot on the PCIe HAT. Ensure the HAT is properly stacked on the Pi 5's GPIO header and secured with standoffs. A loose connection will prevent the card from being detected by the OS.

Steps to verify physical installation:

  1. Power off the Pi completely and disconnect power
  2. Check that the Metis card is fully inserted into the M.2 slot with the retention screw tightened
  3. Verify the PCIe HAT is seated flush on all 40 GPIO pins
  4. Ensure all standoffs are properly secured
  5. Reconnect power and verify detection with lspci | grep -i axelera
§ 02

Installing Raspberry Pi OS

Raspberry Pi Imager is the official tool for writing an OS image to your microSD card. It handles download, writing, and verification in one step. Install it on your desktop or laptop — not the Pi.

Step 1 — Download Raspberry Pi Imager

Download for your OS:

  • Windows: imager_latest.exe — https://downloads.raspberrypi.com/imager/imager_latest.exe
  • macOS: imager_latest.dmg — https://downloads.raspberrypi.com/imager/imager_latest.dmg
  • Linux: run sudo apt install rpi-imager

When asked: "Do you want to allow this app to make changes to your device?" click Yes, After verifying the source

Step 2 — Launch & Select Device

Open Imager. Click Choose Device → select Raspberry Pi 5.

Step 3 — Select Operating System

Click Choose OS → select Raspberry Pi OS (64-bit) from the recommended category.

ℹ️
The 64-bit version is required — the Axelera SDK and Docker images are ARM64 only and will not function on the 32-bit OS. Do not select the Lite or Full variants.

Step 4 — Select Storage

Insert your microSD card to the port. Click Choose Storage and select it from the list.

⚠️
Critical: The write process will completely erase the selected device. Double-check you are not selecting an external drive or USB stick.

Step 5 — OS Customisation

Click Next → Edit Settings and configure:

  • Hostname: e.g. rpi5-voyager
  • Username & password: used for all logins after boot
  • Wi-Fi: enter SSID and password if not using Ethernet
  • Locale: set timezone and keyboard layout (eg: US for English)
ℹ️
Skip the Services tab — SSH is not required for this setup.

Step 6 — Write Image

Click I UNDERSTAND, ERASE AND WRITE and confirm. The Imager will download (~1.1 GB), write, and verify the image. When complete, click Finish and safely eject the card.

Insert the microSD card into the underside slot of the Raspberry Pi 5, then connect all peripherals, connect to the monitor and power it. On first boot the Pi will expand the filesystem and may reboot once — this takes 1–2 minutes and is normal.

From now, all the steps take part in the another Monitor.

⏱️
This may take 10-20 minutes.

§ 04

System Update

Update the package list and upgrade all packages to their latest versions. This ensures the most recent kernel, firmware, and security patches are in place — all of which affect PCIe and driver stability.

bash 
sudo apt update && sudo apt full-upgrade -y

What each part does:

  • apt update — refreshes the local package list from repositories
  • apt full-upgrade -y — upgrades all packages including the kernel; -y auto-confirms

During the upgrade, you may see a prompt asking whether to update or keep the current Chromium configuration. This is part of the system’s package manager—not a browser setting. In most cases, you can safely select the default option (N) to keep your current setup and continue.

Expected output
0 upgraded, 0 newly installed, 0 to remove and 0 not upgraded.
⏱️
This may take 10-30 minutes. Do not reboot after — continue to the next section first.
§ 05

Bootloader & PCIe Configuration

The Pi 5 uses an EEPROM-based bootloader updated separately from the OS. PCIe Gen3 must also be explicitly enabled for full Metis card bandwidth. These require two separate reboots.

Part A — Update the Bootloader

Launch the config tool:

bash
sudo raspi-config

Navigate using the arrow keys and Enter key:

  1. Select Advanced Options → press Enter
  2. Select Bootloader Version → press Enter
  3. Select E1 Latest → press Enter
  4. When asked "Reset boot ROM to defaults?" — select <No> → press Enter
  5. Press Esc to return to the main menu
  6. Use the Tab key to highlight <Finish> → press Enter to exit raspi-config

Apply the update and reboot:

bash
sudo rpi-eeprom-update -a
sudo reboot
ℹ️
If output shows BOOTLOADER: up to date, it's already current. Reboot is still required. Log back in and open a terminal after reboot.

Part B — Enable PCIe Gen3

Gen3 provides significantly higher bandwidth than the default Gen2 and is required for reliable Metis operation.

bash
sudo raspi-config

Navigate using the arrow keys and Enter key:

  1. Select Advanced Options → press Enter
  2. Select PCIe Speed → press Enter
  3. When asked "Would you like to enable PCIe Gen 3 mode?" — select <Yes> → press Enter
  4. Dismiss the confirmation dialog → press Enter
  5. Press Esc to return to the main menu
  6. Tab to <Finish> → press Enter
  7. If asked "Would you like to reboot now?" — select <No> — we will reboot manually below

Reboot to apply the PCIe Gen3 setting:

bash
sudo reboot
§ 06

Install Docker

Docker isolates the Voyager SDK from the host OS, preventing library conflicts. Install Docker Engine via Docker's official Debian repository.

Part A — Prerequisites & GPG Key

bash
sudo apt update
sudo apt install -y ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/debian/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

Part B — Add Docker APT Repository

bash
sudo tee /etc/apt/sources.list.d/docker.sources <<EOF
Types: deb
URIs: https://download.docker.com/linux/debian
Suites: $(. /etc/os-release && echo "$VERSION_CODENAME")
Components: stable
Signed-By: /etc/apt/keyrings/docker.asc
EOF

Part C — Install Docker Engine

bash
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
sudo systemctl enable docker
sudo systemctl start docker

Part D — Post-Install & Reboot

bash
sudo usermod -aG docker $USER
xhost +local:root
xhost +local:docker
sudo reboot

After reboot, verify Docker is working:

bash
docker run hello-world
✅
You should see a message starting with "Hello from Docker!" — Docker is correctly installed.
§ 07

Install Axelera Metis Driver & Runtime

The Metis card requires a kernel driver (DKMS module) and runtime library. These must be installed on the host system — kernel modules cannot be loaded from inside a container.

Part A — Add Axelera GPG Key

bash
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://software.axelera.ai/artifactory/api/security/keypair/axelera/public \
  | sudo gpg --dearmor -o /etc/apt/keyrings/axelera.gpg
sudo chmod a+r /etc/apt/keyrings/axelera.gpg

Part B — Add Axelera APT Repository

bash
sudo tee /etc/apt/sources.list.d/axelera.list <<EOF
deb [signed-by=/etc/apt/keyrings/axelera.gpg] https://software.axelera.ai/artifactory/axelera-apt-source/ ubuntu24 main
EOF

Part C — Install Driver & Runtime

bash
sudo apt update
sudo apt install -y metis-dkms axelera-multi-device axelera-device-1.5.3 \
  axelera-runtime-1.5.3 axelera-riscv-gnu-newlib-toolchain-409b951ba662-7
sudo reboot
⏱️
DKMS will compile the kernel module during install — this takes 3–8 minutes on the Pi 5. Output will scroll; this is expected.

Part D — Verify Driver Installation

After rebooting, open a terminal and run these verification commands one by one:

bash
# Load the module — no output means success
sudo modprobe metis

# Confirm it is active
lsmod | grep metis

# Check DKMS build status
dkms status

# Confirm card is visible on the PCIe bus
lspci | grep -i axelera
Expected output
metis                  131072  0
metis/1.4.16, 6.12.47+rpt-rpi-2712, aarch64: installed
metis/1.4.16, 6.12.47+rpt-rpi-v8, aarch64: installed
metis/1.4.16, 6.12.75+rpt-rpi-2712, aarch64: installed
metis/1.4.16, 6.12.75+rpt-rpi-v8, aarch64: installed
0001:06:00.0 Processing accelerators: Axelera AI Metis AIPU (rev 02)
❌
lspci shows nothing: Card not detected. Re-seat the M.2 card, verify HAT connections, and confirm PCIe Gen3 was enabled in Section 5.
❌
dkms status shows "failed": Module compile error. Run sudo dkms autoinstall to retry and review the output. Ensure kernel headers are installed.
§ 08

Launch Voyager SDK Docker Container

⚠️
Required before every Docker launch: After each reboot, you must run these two commands on the host before starting or entering the container — otherwise display output and inference visualisation will fail.
xhost +local:root
xhost +local:docker

Run the xhost commands first, then launch the container:

bash · host — run after every reboot
xhost +local:root
xhost +local:docker

Then launch an Ubuntu 22.04 container with privileged access to the host's devices and kernel modules so it can communicate with the Metis card:

bash · host — first time only
docker run -it --privileged \
  -v /tmp:/tmp \
  -v /lib/modules:/lib/modules \
  -v /run:/run \
  --device=/dev/dri \
  --device=/dev/vchiq \
  --group-add video \
  -e DISPLAY=$DISPLAY \
  --network=host \
  --name=voyager-sdk \
  ubuntu:22.04
FlagPurpose
-itInteractive TTY — keeps the terminal session open inside the container
--privilegedFull access to host devices and kernel capabilities; required for PCIe device access
-v /lib/modulesShares host kernel modules so the runtime can use the already-loaded metis driver
-v /tmp, /runShared temporary files and runtime sockets (udev) between host and container
--device /dev/driPasses through Direct Rendering Infrastructure for GPU/display access
--device /dev/vchiqPasses through VideoCore hardware interface for Raspberry Pi GPU communication
--network=hostContainer shares the Pi's network stack and IP address directly
--name=voyager-sdkNames the container for easy reference in docker start / docker exec

After the command runs, the terminal prompt changes to:

Container prompt
root@<container-id>:/#
💡
You are now inside the Docker container. All commands from this point forward are run inside the container. To re-enter after a reboot, see the Appendix.
§ 09

Install Voyager SDK Inside the Container

Clone and install the Voyager SDK. The install.sh script handles all Python dependencies, model tools, and runtime bindings.

bash
apt update && apt install -y sudo git pciutils curl lsb-release && update-pciids
cd /home
git clone https://github.com/axelera-ai-hub/voyager-sdk.git
cd voyager-sdk
./install.sh --YES --all --media
  • apt update && apt install ... — installs required packages: sudo, git, pciutils, curl, and updates PCI ID database
  • cd /home — navigates to a stable location that persists for the container's lifetime
  • git clone ... — downloads the full SDK including inference scripts, model configs, and example media
  • --YES — auto-accepts all prompts; --all installs everything including the Python venv, runtime bindings, and model utilities; --media downloads example media files for testing
⏱️
Installation takes 10–30 minutes depending on internet speed. Extensive pip output is expected — this is normal.
❌
If git clone fails: Run curl https://github.com to test connectivity. If it fails, the container was started without --network=host. Exit, run docker rm voyager-sdk, and redo Section 8.
§ 10

Run Inference

With the SDK installed, run AI inference on the Axelera Metis accelerator. Follow these steps in order.

Step 1 — Activate the Virtual Environment

bash
source venv/bin/activate

Your prompt will change to show (venv) at the start. Re-run this each time you start a new container session.

Step 2 — Verify the Accelerator is Detected

bash
axdevice
Expected output
Axelera Metis Device 0
  PCI address : 0001:01:00.0
  Status      : Ready
  Firmware    : v1.5.x

Step 3 — Explore Options (Optional)

bash
./inference.py --help

Step 4 — Run a Test Inference

bash
./inference.py yolov5s-v7-coco media/traffic1_1080p.mp4

This runs YOLOv5s object detection (COCO dataset, 80 classes) on the bundled 1080p traffic video, drawing bounding boxes and printing real-time FPS.

Expected output
Loading model: yolov5s-v7-coco
Compiling for Metis AIPU...
Running inference on: media/traffic1_1080p.mp4
FPS: 45.3  |  Frame: 120/600  |  Objects detected: 8
FPS: 46.1  |  Frame: 240/600  |  Objects detected: 5
...
Inference complete. Average FPS: 45.7
✅
Success: FPS is displayed and incrementing — the Metis card is offloading and accelerating inference. Expected range is 30–60+ FPS for YOLOv5s at 1080p. Lower FPS may indicate thermal throttling — check vcgencmd measure_temp.

Step 5 — If Inference Fails: Reboot Recovery

If inference crashes, hangs, or axdevice returns no devices after a previously working session, the AIPU may be in a bad state. A full reboot sequence clears this reliably.

⚠️
Exit the container before rebooting — type exit in the container shell to return to the host, then proceed with the steps below.

On the host — reboot and restore:

bash · host
# Step 1 — Reboot the Pi
sudo reboot

After the Pi comes back up, open a terminal and run:

bash · host — after reboot
# Step 2 — Reload the kernel module
sudo modprobe metis

# Step 3 — Grant display access (required every boot)
xhost +local:root
xhost +local:docker

# Step 4 — Restart the container
docker start voyager-sdk

# Step 5 — Re-enter the container
docker exec -it voyager-sdk bash

Inside the container — resume inference:

bash · container
cd /home/voyager-sdk
source venv/bin/activate
axdevice  # verify card is ready
./inference.py yolov5s-v7-coco media/traffic1_1080p.mp4
ℹ️
If axdevice still shows no device after a reboot, run lspci | grep -i axelera on the host to confirm the card is visible at the PCIe level. If it is not, power-cycle the Pi fully (not just reboot) and re-check physical connections.

📎 Appendix & Quick Reference

Every Boot Checklist

After any Pi reboot, always run these steps on the host before entering the container:

bash · host — every boot
# 1. Reload the AIPU kernel module
sudo modprobe metis

# 2. Grant display access to Docker
xhost +local:root
xhost +local:docker

# 3. Start the container
docker start voyager-sdk

# 4. Enter the container
docker exec -it voyager-sdk bash

Then inside the container:

bash · container
cd /home/voyager-sdk
source venv/bin/activate

Re-entering the Container After a Reboot

After a Pi reboot, the container will be stopped. Use these steps to get back in:

Step 1 — Load the kernel module on the host:

bash · host
sudo modprobe metis

Step 2 — Grant display access (required every boot):

bash · host
xhost +local:root
xhost +local:docker

Step 3 — Start the stopped container:

bash · host
docker start voyager-sdk

Step 4 — Attach an interactive shell into the running container:

bash · host
docker exec -it voyager-sdk bash

Step 5 — Inside the container, re-activate the environment:

bash · container
cd /home/voyager-sdk
source venv/bin/activate
ℹ️
docker start brings the container back from stopped state. docker exec -it ... bash opens a new interactive shell inside it. This is different from the original docker run — that command only works the first time to create the container.

Useful Docker Commands

bash
# List all containers (running and stopped)
docker ps -a

# View container logs
docker logs voyager-sdk

# Remove the container entirely to start fresh
docker rm voyager-sdk

❓ Frequently Asked Questions

Common questions from first-time setup and everyday use.

Setup Do I need to run xhost commands every time I reboot? ▶

Yes. xhost +local:root and xhost +local:docker are not persistent — they reset on every reboot. You must run both on the host before starting or entering the container, or the inference visualisation window will fail to open.

A quick way to remember: treat them as part of your startup sequence alongside sudo modprobe metis and docker start voyager-sdk.

Docker What happens if I accidentally use docker run again instead of docker start? ▶

docker run tries to create a new container. Because a container named voyager-sdk already exists, Docker will throw an error: Conflict. The container name "/voyager-sdk" is already in use.

This is harmless — your existing container and all installed SDK files are untouched. Simply use docker start voyager-sdk and then docker exec -it voyager-sdk bash to get back in as normal.

Inference Inference worked before but now "axdevice" shows no devices — what happened? ▶

This usually means the kernel module was not reloaded after a reboot, or the AIPU entered a fault state. Fix it with the reboot recovery sequence in Section 10 Step 5:

  1. Exit the container: exit
  2. On the host: sudo reboot
  3. After reboot: sudo modprobe metis, then xhost +local:root and xhost +local:docker
  4. Restart and re-enter the container, then check axdevice again

If the card still doesn't appear, verify it's visible at the PCIe level: lspci | grep -i axelera on the host. If nothing shows there, power-cycle the Pi fully.

Hardware The Pi crashes or reboots randomly during inference — what's wrong? ▶

Most likely cause: insufficient power. The Metis AIPU draws significant additional power via PCIe. A standard 15W (5V/3A) supply is not enough. You must use a 27W (5V/5A) USB-C PSU — this is listed as a requirement in Section 1.

Secondary causes to check:

  • Thermal throttling or shutdown — check vcgencmd measure_temp. Over 80°C is concerning.
  • Poor quality or slow microSD card causing filesystem errors under load
  • Loose M.2 card or HAT connection — re-seat the card with the Pi powered off
Inference FPS is much lower than the expected 30–60+ — how do I diagnose this? ▶

Run these checks in order:

  1. Temperature: vcgencmd measure_temp on the host. If over 80°C the Pi is throttling. Improve cooling.
  2. PCIe speed: Confirm Gen3 is active — sudo cat /sys/bus/pci/devices/0001:01:00.0/current_link_speed should show 8.0 GT/s PCIe. If it shows Gen1 or Gen2, re-run Section 5.
  3. Card status: axdevice should show Status: Ready. Any other status means the card needs a reboot recovery.
  4. CPU load: Run htop on the host in a separate terminal. High CPU usage unrelated to inference can compete for shared resources.
Docker Will reinstalling Docker or removing the container delete my SDK installation? ▶

Yes — running docker rm voyager-sdk will permanently delete the container, including the Voyager SDK cloned and installed inside it at /home/voyager-sdk. You would need to repeat Section 9 from scratch.

Only do this if you need a completely clean slate. For most problems (bad state, display issues, inference failures) a reboot and container restart — without removing the container — is sufficient.

Uninstalling Docker from the host does not immediately delete stopped containers, but removing Docker's storage directory will. Treat the container as your persistent SDK environment and avoid unnecessary removal.

Network Can I SSH into the Pi instead of using a keyboard and monitor? ▶

Yes, with limitations. SSH works fine for all command-line steps. However, inference visualisation requires a display — the inference window renders via X11 directly to the screen. If you're running headless over SSH, you won't see the output window unless you set up X11 forwarding (ssh -X) to a machine with a display server.

To enable SSH: in Raspberry Pi Imager's OS customisation (Section 2, Step 5), go to the Services tab and enable SSH. You can then connect via ssh username@hostname.local. But, Wi-Fi or Network details entered at Step-5 should be the same that is used by the laptop.

Setup How do I check how much disk space is left on the Pi? ▶

Run df -h / on the host to see filesystem usage. The Voyager SDK installation (including models and media) takes 3–6 GB. If you're running low on space, check for large log files or unused Docker images:

  • docker images — lists downloaded images
  • docker image prune — removes unused images to reclaim space
  • du -sh /home/voyager-sdk — shows SDK directory size inside the container

A 32 GB microSD card is the minimum recommended; 64 GB gives more comfortable headroom.

Hardware The guide says 20–50 minutes — what takes the longest? ▶

The main time sinks are all download and compile operations:

  • Section 4 (system update): 5–15 min depending on how many packages need upgrading
  • Section 7 (DKMS kernel module compile): 3–8 min — this is CPU-bound and cannot be sped up
  • Section 9 (Voyager SDK install): 10–30 min — almost entirely internet speed dependent

On a fast connection (100 Mbps+) with a modern microSD, the full process is typically done in around 25–30 minutes. On a slower connection or congested network it can stretch to 50 minutes.
